# ADR-0004：分离沙箱强制边界与审批策略

- 状态：已接受并实施，真实 Windows 强制边界待验收
- 日期：2026-06-09

## 背景

模型生成的命令或文件操作可能错误、具有破坏性，或受到不可信仓库内容影响。
只有用户确认并不能形成技术边界，而只有沙箱也无法表达用户意图。

Mimodex 还需要在允许调用 MiMo API 的同时，让 Agent 命令默认保持离线。

## 决策

将沙箱强制边界与审批策略实现为两个独立、可组合的控制：

- 沙箱定义文件系统、进程和网络操作在技术上是否可发生。
- 审批策略决定尝试越界时，是阻止、自动允许，还是提交用户审批。
- Provider 网络与 Agent 命令网络相互独立。
- 所有强制检查在工具执行前由 Runtime 完成，不能只依赖桌面 UI。

首版提供只读、工作区写入和完全访问模式。完全访问需要用户明确选择并看到醒目警告。

## 实施状态

桌面端已提供三种权限模式、完全访问高风险确认、持续权限标识和包含工作目录、授权
边界与网络需求的审批卡片；Runtime 继续负责实际沙箱和审批强制。真实 Windows 11
权限边界、命令联网和进程树取消仍属于私测准入验收项。

## 决策理由

- 只有当未获支持的动作仍被技术阻止时，用户同意才有意义。
- 独立网络能力可避免 API 访问无意间赋予 Shell 命令联网权限。
- Runtime 统一强制执行可覆盖当前和未来所有客户端。

## 影响

### 正面影响

- 威胁模型和用户预期更加清晰。
- 降低 Prompt 注入与意外破坏性命令的风险。
- 审批决定可以审计和重放。

### 负面影响

- Windows 与 macOS 的原生沙箱行为不同。
- 进程树取消和子进程限制需要平台专项测试。
- 某些开发流程需要明确审批或完全访问模式。

## 首版必须识别的受保护动作

Runtime 至少必须识别：

- 写入用户选定项目之外；
- 修改 `.git` 等受保护项目元数据；
- 破坏性删除或覆盖；
- 命令联网；
- 安装软件或修改系统配置；
- 访问凭据存储；
- Git push 或其他远程修改；
- 启动脱离管理的后台进程。

## 未采用方案

### 只有确认弹窗，没有沙箱

不采用。分类遗漏或 UI Bug 都可能直接放行动作。

### 只有沙箱，没有审批

不采用。合法越界操作需要清晰、明确的用户同意路径。

### Provider 网络开启时自动允许命令联网

不采用。两种能力的目的和风险不同。
