# ADR-0005：在本地持久化可重放的 Agent 线程

- 状态：已接受
- 日期：2026-06-09

## 背景

Mimodex 线程必须能在应用重启后恢复，并保存足够的 Provider 历史以支持多轮
工具调用。MiMo 可能要求将之前的助手 `reasoning_content` 与工具调用历史一同
回传。只保存 UI 渲染后的聊天文本，无法安全恢复 Agent 循环。

产品还需要保留审批、命令和文件变更的审计记录。

## 决策

在本地持久化权威、有序、可重放的线程记录，其中包括：

- 用户和助手消息；
- Provider 协议要求的助手推理内容；
- 工具调用参数、ID、结果与状态；
- 审批请求与决定；
- 模型、Provider、权限和请求元数据；
- 轮次完成、失败、中断和取消标记。

使用只追加持久记录作为事实来源，并使用 SQLite 查询投影提高桌面端查询速度。
凭据继续独立保存在操作系统凭据存储中。

## 决策理由

- 支持正确重放 MiMo 请求。
- 让线程恢复和崩溃恢复具有确定性。
- 提供可审阅的 Agent 动作记录。
- 先追加后确认的存储方式降低轮次被错误标记完成或部分覆盖的风险。

## 影响

### 正面影响

- 可靠的恢复与审计能力。
- UI 状态可以从持久事件重建。
- 更容易调试 Provider 与工具协议失败。

### 负面影响

- 推理与工具记录可能包含敏感项目内容。
- 必须设计保留、删除、导出与脱敏策略。
- 必须支持 Schema 演进和迁移。
- 长线程需要在不违反 Provider 重放规则的前提下压缩。

## 隐私与保留规则

- 默认仅本地存储。
- API 凭据永不写入线程记录。
- 为用户提供清晰的归档和永久删除线程能力。
- 日志与权威会话记录分离，且默认脱敏。
- 不将会话数据作为遥测发送。
- 明确说明哪些项目内容会被发送到配置的 MiMo 端点。

## 崩溃恢复规则

- 只有完成标记持久化后，轮次才算完成。
- 重启时，没有完成标记的活动轮次被标记为已中断。
- 副作用状态不确定的工具调用不得自动重试。
- 重复流式事件必须能够安全去重。

## 未采用方案

### 只保存渲染后的聊天文本

不采用。该方案会丢失正确恢复所需的推理、工具身份、审批和协议状态。

### 只使用 SQLite 可变行作为事实来源

不采用。与有序持久记录相比，部分更新和问题调试更加困难。

### 默认云同步会话记录

首版不采用。该方案与本地优先定位冲突，并增加账号、隐私和安全范围。
