## 1. 验证目标

证明首版默认模型 MiMo v2.5 能可靠驱动本地编程 Agent 闭环，验证高级模型
MiMo v2.5 Pro 的兼容性，并明确接入分支后 Codex Runtime 所需的 Adapter 改动。

本次 Spike 是一次可丢弃的技术调查，不是桌面端 MVP。优先编写小型协议探针和
测试，不要提前构建生产级抽象。

## 2. 强制交付物

- 使用真实 MiMo API 的最小命令行探针；
- 每个强制场景对应的已脱敏请求与流式响应 Fixture；
- `mimo-v2.5-pro` 与 `mimo-v2.5` Provider 能力矩阵；
- Codex Runtime 项目/事件与 MiMo wire 消息之间的映射说明；
- 基于选定 Codex 上游 commit 的必需修改清单；
- 对照退出条件的通过/失败报告；
- “继续、调整协议或停止”的最终建议。

## 3. 安全规则

- 尽可能使用预算受限的专用测试 API 凭据。
- 不得提交凭据、未脱敏提示词、私有仓库内容或包含敏感信息的原始日志。
- 文件与 Shell 工具必须使用合成 Fixture 仓库。
- 探针工具只能使用允许列表，并在可丢弃目录中运行。
- 纯协议测试不得执行模型直接生成的原始 Shell 命令。

## 13. 停止条件

如果出现以下任一情况，暂停当前架构路线并撰写决策备忘录：

- MiMo 无法在进程重启后可靠重放工具型对话。
- 推理重放规则无法与安全上下文压缩共存。
- 流式工具调用身份或参数无法可靠重建。
- Provider 错误导致副作用重试安全性无法判断。
- 接入 Chat Completions 需要侵入式修改大量无关 Runtime 层。
- 必需 API 行为在账号或端点间存在实质差异，且没有稳定兼容契约。

可能的处理方案包括：

- 评估 MiMo Anthropic 兼容 API；
- 放弃 Codex 分支，使用更窄的自定义 Runtime；
- 缩减首版 Agent 能力；
- 在 Provider 行为明确前暂停项目。
